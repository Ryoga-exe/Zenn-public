## Q1(共通)

### Q1.1

> プロセッサゼミをなぜ志望するか教えてください。あわせて、本ゼミで取り組みたいことがあれば記述してください。ゼミではまず基本的な課題に取り組んだ後、受講者の皆さんそれぞれの興味に応じた発展課題に取り組む時間を設ける予定です。
> また、何か他にアピールしたいこと（今まで作ってきたものなど）があれば、それも自由に書いてください。

#### 回答

プロセッサゼミを志望する理由は、自分で RISC-V CPU を実装した経験を通して、プロセッサ設計についてより体系的に学びたいと感じたからである。
特に、以前実装した CPU では「正しく動作させること」を優先したため、性能改善や高速化に十分取り組むことができなかった。
そのため、本ゼミでプロセッサ設計の基礎を改めて学び、より性能を意識した設計ができるようになりたいと考えている。

以前、ChiffonCore という RISC-V CPU を実装した。
この CPU では、Ultra96-V2 ボード上に RV64IMAC_Zicsr_Zicntr 相当の CPU を実装し、UART やメモリマップド I/O、描画コマンドを扱う IP などと組み合わせて、自作 OS を動かすところまで取り組んだ。
ソースコードは GitHub 上で公開しており、実際にこの CPU 上で自作 OS を動かしている様子も動画として記録している。

GitHub: https://github.com/Ryoga-exe/ChiffonCore
動作動画:
https://youtu.be/sns4nfxTDY8

この経験を通して、命令デコーダ、パイプライン、CSR、例外処理、メモリアクセス、周辺回路との接続など、プロセッサを構成する各要素がどのように関係しているのかを、実装を通じて学ぶことができた。
一方で、まずは CPU として正しく動作することを重視していたため、性能面については十分に検討できなかった。
実際に、この CPU 上でベンチマークとして CoreMark を動作させたところ、FPGA の動作周波数に対してかなり低いスコアとなり、マイクロアーキテクチャの設計やメモリアクセス、パイプライン制御などに改善の余地が大きいことを実感した。

本ゼミで取り組みたいこととしては、まず基本的な課題を通して、プロセッサの設計をより理論的・体系的に理解したい。
その上で、発展課題では、以前実装した ChiffonCore の経験を踏まえ、CPU の性能改善や検証環境の整備に取り組みたいと考えている。
また、性能を改善するだけでなく、CoreMark などのベンチマークを用いて、変更前後でどの程度性能が変化したのかを定量的に評価できるようにしたい。

アピールしたいこととして、私はこれまで、単に CPU の一部だけを実装するのではなく、CPU コア、周辺回路、シミュレーション環境、実機動作、自作 OS の実行まで一通り取り組んできた経験がある。

また、CPU や OS の実装といった比較的低いレイヤにとどまらず、大学の新歓期間のための Web サイト (https://shinkan-web.zdk.tsukuba.ac.jp/, 記事: https://zenn.dev/ryoga_exe/articles/tsukuba-shinkan-web) の実装や、ゲーム開発にも取り組んできた。
これらの経験を通して、アイデアを実際に動く形のある成果物としてアウトプットする力を身につけてきた。
私の強みは、興味を持った技術について、調べるだけで終わらせず、実際に動くものとして実装し、公開・発信するところまで取り組める点である。
また、これらの開発ではチームで設計や実装を進め、他のメンバーと議論しながら仕様や実装方針を決める経験もした。
この経験は、本ゼミにおいて講師や他の参加者と議論しながら課題に取り組む上でも活かせると考えている。

#### 感想

### Q1.2

> ターミナルでcoding agentに「このディレクトリ以下の構造を把握して」とプロンプトを入力してから結果がターミナルに表示されるまでにコンピュータの内側で起こっていることを、プロセッサの視点に立って説明してください。

#### 回答

回答作成の調査にあたって検索などの他に OpenAI Codex CLI (https://github.com/openai/codex) のコードの観察も行った。
この観察結果で得られた具体的な実装も交えて回答する。

プロセッサは「日本語の意味」や「AI の意図」を直接理解しているわけではない。これは強調しておきたいと思っている。
プロセッサから見えているのは、メモリ上のビット列、レジスタの値、実行すべき機械語命令、OS から渡される割り込みやシステムコールの結果である。
ユーザから見ると「指示を入力したら agent が考えた」ように見えるが、プロセッサから見ると、それは命令実行、メモリアクセス、I/O 待ち、プロセス切り替えの連続である。

まず、ユーザがキーボードで文字を入力する。
キーが押されると、キーボードや OS の入力処理を通じてイベントが発生する。
プロセッサは通常、別の命令列を実行しているが、入力イベントが来ると OS の管理下で割り込み処理に入る。
プロセッサは現在のレジスタ状態などを退避し、OS カーネル内の入力処理へ移る。

入力されたキー情報は、OS やターミナル、必要なら IME によって文字データへ変換される。
日本語入力の場合、k, o, n, o のようなキーボードの入力列が、変換候補の選択や確定を経て「この」という文字列になる。
プロセッサはその間、IME やターミナルアプリの命令を実行し、入力中の文字列をメモリ上のバッファに置いていく。

ターミナルは単なる画面ではなく、OS 上で動く一つのアプリケーションプロセスである。
ターミナルは入力された文字を画面に描画しつつ、裏側では仮想的な端末装置である PTY を通して、シェルや coding agent に入力を渡す。

ユーザが Enter を押すと、入力行が確定する。
coding agent がすでに起動していて入力待ちの状態なら、その一行は標準入力から agent プロセスへ届く。
多くの場合、agent 側は read のようなシステムコールで入力が来るまで待つ状態になっている。
入力が届くと、OS はそのプロセスを実行可能状態に戻し、スケジューラが順番を決める。
そしてプロセッサは再び coding agent の命令列を実行し始める。

coding agent も普通のプログラムである。
コードが公開されているものである OpenAI Codex CLI を例にする。(https://github.com/openai/codex)
リポジトリの `codex-rs/README.md` では Rust 実装が現在の保守対象の CLI であり、デフォルトの体験だと説明されている。
また、同じ README には codex exec PROMPT や標準入力からプロンプトを渡せること、出力がターミナルに直接表示されることも書かれている。
つまり「ターミナルでプロンプトを受け取り、処理し、結果を端末へ返す」という課題の状況は、実際の Codex CLI の構造と対応していることがわかる。

Codex の TUI 側を見ると、`codex-rs/tui/src/bottom_pane/chat_composer.rs` にはキー入力を処理する `ChatComposer` があり、Enter などの submit キーが押されると投稿処理へ進む。
さらに `codex-rs/tui/src/chatwidget/input_submission.rs` では、入力されたテキストが `UserInput::Text` のようなデータに変換され、`AppCommand::user_turn` として agent の 1 ターンに渡される。
プロセッサから見ると、これは「文字列を構造化データに詰め替え、次の処理へ渡す Rust の命令列を実行している」だけである。

agent は「このディレクトリ以下の構造を把握して」という入力を受け取ると、そのタスクを達成するためには、現在の作業ディレクトリ以下のファイルやディレクトリを調べる必要があると判断するであろう。
また、この判断は、ローカルの LLM で実行されるか、外部の LLM に問い合わせて実行される。
外部モデルを使う場合、agent はユーザ入力、現在の作業ディレクトリに関する情報、利用可能なツール、過去の会話、システム指示などをまとめ、HTTP リクエストとして送る。
HTTP 通信を行う際、agent のプロセスは socket、connect、send、recv などに相当する処理をライブラリやシステムコールを通じて OS カーネルに依頼する。
この過程で CPU はユーザモードとカーネルモードを行き来する。
通信待ちが発生すると、ブロッキング I/O の場合、そのスレッドやプロセスは待ち状態になり、OS のスケジューラはその間に別のプロセスやスレッドを実行できる。
ノンブロッキング I/O や非同期ランタイムを使う実装では、待機中の処理だけを中断し、同じプロセス内で別のタスクを進めることもある。

次に agent は、ディレクトリ構造を知るために実際の観測を行う。
たとえば `pwd` で現在位置を確認し、`ls`、`find`、`tree` などでファイル一覧を取るかもしれない。
これは「AI が雰囲気で把握する」のではなく、ファイルシステムに問い合わせて、実際のディレクトリ名、ファイル名、設定ファイルの内容を集める処理である。

Codex の実装にも、この抽象化が見える。
`codex-rs/file-system/src/lib.rs` には `ExecutorFileSystem` という `trait` があり、`read_file`、`read_file_text`、`get_metadata`、`read_directory` などの操作が定義されている。
これは、agent の各部品が「ファイルを読む」「メタデータを取る」「ディレクトリを列挙する」という操作を、ローカル環境でもリモート実行環境でも同じ形で呼び出せるようにする層である。
さらに `codex-rs/exec-server/src/local_file_system.rs` では、ローカルファイルシステムに対して `tokio::fs::read`、`tokio::fs::metadata`、`tokio::fs::read_dir` などを使っている。
プロセッサ視点では、Rust の非同期処理を進めながら、最終的には OS のファイル読み取り機能へ降りていく。

ファイルシステムを読むとき、ユーザプログラムである agent はストレージを直接操作することはない。
agent は open、read、stat、getdents、close に相当するシステムコールを通じて OS に依頼する。
プロセッサはシステムコール命令を実行してユーザモードからカーネルモードへ切り替わり、OS のファイルシステム処理を実行する。
OS は命令に応じて「このパスは存在するか」「これはファイルかディレクトリか」「中にはどんなエントリがあるか」などを処理する。

必要な情報がすでにメモリ上のページキャッシュやディレクトリエントリキャッシュにあれば、ストレージにアクセスせずに結果が返る。
キャッシュにない場合、OS は SSD や HDD に読み取り要求を出す。
ストレージ I/O はプロセッサの演算に比べて非常に遅いので、その間 agent プロセスは待機状態になることがある。
読み取りが完了すると I/O 完了割り込みが発生し、OS は待っていたプロセスを再び実行可能にする。
プロセッサはスケジューラに従って agent の処理を再開する。

agent がシェルコマンドを使う場合は、さらに別のプロセスが起動される。
たとえば agent が grep -r を実行するなら、OS は新しいプロセスを作り、その実行ファイルをメモリにマップし、標準出力を pipe や PTY につなぐ。
プロセッサは今度は grep の機械語命令も実行し、grep はディレクトリを再帰的に読みながらファイル一覧を出力する。
その出力は OS のバッファを通って agent に戻る。
Codex でも `codex-rs/core/src/tools/handlers/shell.rs` に、コマンド実行要求を `ShellRuntime` に渡し、結果を tool output として扱う流れが観察された。

集められたファイル一覧や設定ファイルの内容は、agent のメモリ上で解析される。プロセッサは文字列比較、パスの分割、JSON や TOML のパース、配列やハッシュマップへの格納、条件分岐などを高速に繰り返す。
プロセッサ自身が意味を理解しているわけではなく、プログラムに書かれた規則とモデルの出力に従ってデータを処理している。

外部 LLM を使っている場合、agent は集めた情報を再びプロンプトにまとめてサーバーへ送る。
サーバ側では別のプロセッサや GPU がトークン化、行列演算、次トークン予測を行い、応答文を生成する。
ローカルモデルを使う場合は、同じような推論計算が手元の CPU や GPU で行われる。
どちらの場合も、最終的には、次にどの文字列を返すかというデータが agent プロセスに戻る。

回答ができると、agent は標準出力へ文字列を書き込む。
これは write に相当するシステムコールで行われる。
標準出力は PTY につながっているため、出力データは OS を通じてターミナルアプリへ渡される。
ターミナルは受け取った文字列を解釈し、必要なら ANSI エスケープシーケンスによる色やカーソル移動も処理する。
その後、文字の位置を決め、OS のウィンドウシステムや GPU を通じて画面に描画する。

以上をまとめると、プロセッサはまずキーボード入力によるイベントを処理し、ターミナルと PTY を通して文字列を coding agent に届ける。
次に agent のプログラムを実行し、入力を構造化し、必要に応じて LLM と通信し、ファイルシステムやシェルコマンドを使ってディレクトリ構造を観測する。
その間、プロセッサはシステムコール、割り込み、プロセス切り替え、メモリアクセス、キャッシュ利用、I/O 待ちを繰り返す。
最後に agent の生成した回答を標準出力へ書き込み、ターミナルがそれを画面に表示する。

#### 感想

単に回答するだけでなく、実際のコードとかについても触れるといいのかなと思ったため、Codex のコードを適宜参照して回答しました。
ただ、少しまとまりの無い回答になってしまったようにも感じます。

## Q2 (選択・CPU自作)

> Q2は難しい問題が多いので、すべてを完璧に回答できる必要はありません。調べても分からないことがあれば「どこまで調べて、何が分からなかったか」を書いてください。
> Q2.2、Q2.3、Q2.4は、CPUが命令を処理する仕組みと構造の観点から回答してください。

### Q2.1

> CPU自作班では受講者がそれぞれ好きなISAのCPUを実装し、高速化や機能の追加に取り組んでもらう予定です。あなたが実装したいCPUの種類、使用するプログラミング言語、ハードウェア記述言語を教えてください。講師が予習するためにも、詳しく教えてください。

#### 回答

私は RISC-V ISA の CPU を実装したいと考えている。
特に、最終的には RV64GC の実装を目標にしたい。
RV64GC は 64bit の RISC-V アーキテクチャであり、それに加えて以下の拡張を実装したものである。

- I （基本整数命令）
- M （乗除算命令）
- A （アトミック命令）
- F （単精度浮動小数点命令）
- D （倍精度浮動小数点命令）
- C （16ビットの圧縮命令）

実装には、ハードウェア記述言語として Veryl を使用したい。(https://veryl-lang.org/)
Veryl は SystemVerilog に近い構文を持ちながら、より安全で書きやすい HDL として設計されているため、CPU のような比較的大きな回路を構造化して実装するのに向いていると考えている。

プログラミング言語については、テストやシミュレーション補助、ツール作成のために Python や Zig を使用したい。
特に、命令デコーダの検証、テストケース生成、シミュレーション結果の確認などに活用したいと考えている。
Zig (https://ziglang.org/) は、C 言語の相互運用が容易であり、また C/C++ を含むクロスコンパイラとして簡単に利用できるため使用するつもりである。
特に Zig で実装するというよりかは、テストケース作成においての C 言語のクロスコンパイラとして利用しようと考えている。

#### 感想

「詳しく教えてください」とあったので、RISC-V の拡張についてそれぞれ軽く説明を入れました。
また、Veryl + Zig については、以前取り組んだ自作 CPU で使ったことがあり、体験が良かったので今回も使いたいなという気持ちがありました。

### Q2.2

> 同じプログラムを実行した場合であっても、昔の高性能なCPUと比較して、最新の高性能なCPUの方が実行時間が基本的に短くなっています。何が差異を生み出しているのでしょうか？比較する年代、技術や手法を明示したうえで、CPUで命令が実行される手順に注目して説明してください。

#### 回答

CPI を、1 命令あたり何クロックかかるかとすると、実行時間は大まかに、

実行時間 = 実行命令数 × CPI ÷ クロック周波数

で考えられる。

つまり、同じプログラムを実行した場合、クロック周波数が高いほど実行時間は短くなり、CPI が小さいほど実行時間も短くなる。

最新の CPU の高速さは、クロック周波数の増大のみならず、内部で命令を実行する手順の各段階の改良による、CPI が小さくなっていることにも要因がある。

CPU は、命令をメモリからフェッチし、その命令をデコードし、内部で扱いやすい形に変換し、依存関係を確認し、実行ユニットに投入し、最後に結果を確定する。
この一連の手順のそれぞれで、昔の CPU と最新 CPU には大きな差がある。

先にまとめを提示する。

同じプログラムを実行しても、最新 CPU の方が速い理由は次のように整理できる。
まず、クロック周波数が高くなったことで、同じ CPI でも実行時間は短くなる。
しかし、同様に、CPU 内部の命令実行手順が進化したことも理由にある。

- 命令フェッチでは、分岐予測によって次に必要な命令を先に取りに行く。
- デコードでは、複雑な命令を μOp などの命令変換を行い、必要に応じて融合し、Op Cache で再利用する。
- レジスタ・リネーミングでは、見かけ上の依存を消し、独立した命令を並列に実行しやすくする。
- スケジューリングでは、依存していない命令を先に実行する。
- 実行段階では、多数の実行ユニットにより、複数の命令を同時に処理する。
- メモリアクセスでは、多段キャッシュやプリフェッチによってメモリ待ちを減らす。
- 投機実行では、分岐結果が確定する前に予測に基づいて先へ進む。

したがって、昔の CPU は、命令をできるだけ速く順番に流すことが中心だったのに対し、最新 CPU は、同じ結果になる範囲で、命令を先読みし、分解し、並べ替え、複数同時に実行することで高速化している。

その結果、同じプログラムであっても、最新 CPU では CPI が小さくなる。
これが、実行時間の差を生み出していると説明できる。

以下に CPU で命令が実行される手順に分けて説明する。
ここでは、1990 年代前半の Intel 486 DX2、1990 年代半ばの Pentium Pro、2000 年代以降の Intel Core 系、2020 年代の AMD Zen 5 および Intel Lion Cove 世代を比較対象とした。
また、適宜カッコ内に参照したサイトや、データシートなどを示している。

##### ＜命令フェッチ＞

最初に CPU は、次に実行すべき命令をメモリまたはキャッシュから取り出す。

Intel 486 DX2 の時代にも、8KB のオンチップキャッシュやパイプライン処理があり、当時としては高性能だった。
多くの命令を 1 クロックで実行できるようにする設計も採られていたようだ。(https://ardent-tool.com/CPU/docs/Intel/486/datasheets/271293-002.pdf)

しかし、現代 CPU では、命令フェッチの前に分岐予測が働く。
if 文やループのような分岐に対して、次にどちらへ進むかを予測し、予測した先の命令を先に取りに行く。

古い CPU では、分岐の結果がわかるまで待つという状況が多かったが、最新の CPU では、予測が当たれば待ち時間をほぼ隠すことができる。
これにより、パイプラインが空になる時間が減り、CPI が小さくなることに繋がる。

AMD Zen 5 では、TAGE 系の分岐予測、大きな BTB、命令フェッチ帯域の拡大、Op Cache などが導入されており、命令供給を途切れにくくしている。(https://hc2024.hotchips.org/assets/program/conference/day2/24_HC2024.AMD.Cohen.Subramony.final.pdf)

##### ＜命令デコード＞

フェッチされた命令は、そのまま実行されない。
特に、x86 命令は形式が複雑であるから、CPU 内部では μOp と呼ばれる小さな内部命令に変換される。

Intel 486 のような古い CPU では、命令を比較的素直にパイプラインへ流していた。
一方、Pentium Pro 以降では、命令を内部命令に分解し、それらを並べ替えたり、複数同時に実行したりする設計が本格化した。(https://datasheets.chipdb.org/Intel/x86/Pentium%20Pro/24269001.pdf)

Intel Core 系では、複数のデコーダによって複数命令を同時にデコードできる。
また、micro-fusion や macro-fusion によって、複数の処理を少ない μOp として扱える場合がある。(https://www.intel.com/content/dam/doc/manual/64-ia-32-architectures-optimization-manual.pdf)

さらに現代 CPU には、デコード済みの μOp を保存する Op Cache がある。
同じループを何度も実行する場合、毎回デコードし直さずに済むため、命令供給が速くなる。

順に解釈して実行へ送るものから、命令を内部命令に分解し、必要なら融合を行い、再利用し、複数まとめて実行準備へ送るというものに進化している。

##### ＜レジスタ・リネーミング＞

プログラム上では、同じレジスタ名が何度も使われるが、同じレジスタ名を使っているからといって、すべての命令が本当に依存しているとは限らない。

現代 CPU は、レジスタリネーミングによって、プログラムから見えるレジスタ名を内部の物理レジスタに割り当て直す。これにより、見かけ上の依存関係を取り除くことができる。

a = b + c と d = e + f という計算が互いに無関係であるならば、CPU はこれらを同時に進められるわけである。

Intel Lion Cove では、割り当て・リネーム幅やリタイア幅が拡大され、より多くの命令を同時に処理できるようになっている。(https://pc.watch.impress.co.jp/docs/column/silicon_exp/1658790.html)

##### ＜スケジューリングとアウトオブオーダ実行＞

古い CPU では、命令は基本的にプログラムに書かれた順序に近い形で実行される。そのため、ある命令がメモリアクセスで止まると、後ろの命令も待たされやすい。

現代 CPU はアウトオブオーダ実行を行う。これは、プログラムの意味を変えない範囲で、実行できる命令から先に実行するものである。

Pentium Pro では、分岐予測、データフロー解析、投機実行を組み合わせた Dynamic Execution が導入された。これは、命令を単純に順番通り実行するのではなく、依存関係を見て実行順序を内部で最適化する考え方である。(https://atmarkit.itmedia.co.jp/fsys/pcencyclopedia/009procs_hist03/procs_hist06.html, https://tams.informatik.uni-hamburg.de/lehre/2001ss/proseminar/mikroprozessoren/papers/pentium-pro-performance.pdf)

これによって、メモリの待ちや、一部の演算待ちの間にも、別の命令を進めることができる。
結果として、CPU が何もしていない時間が減り、CPI が小さくなる。

##### ＜実行ユニット＞

命令が実行段階に入ると、整数演算、浮動小数点演算、アドレス計算、ロード、ストア、分岐処理などを担当する実行ユニットに送られる。

Intel 486 では、多くの命令を 1 クロックで実行できること自体が大きな特徴だった。(https://ardent-tool.com/CPU/docs/Intel/486/datasheets/271293-002.pdf)

しかし現代 CPU では、1 クロックで 1 命令を実行するだけではなく、複数の μOp を複数の実行ユニットで同時に処理する。

AMD Zen 5 では、8 命令幅 の dispatch、rename、retire、複数の整数 ALU、AGU、分岐ユニット、広いベクトル実行器などが示されている。(https://hc2024.hotchips.org/assets/program/conference/day2/24_HC2024.AMD.Cohen.Subramony.final.pdf)

Intel Lion Cove でも、実行ポート数の増加、整数演算器や分岐処理能力の強化、リタイア幅の拡大などが行われている。(https://pc.watch.impress.co.jp/docs/column/silicon_exp/1658790.html)

すなわち、1 クロックで 1 つ、または少数の処理を進めるから、1 クロックで複数の命令由来の処理を同時に進めるという進化がある。
この違いにより、同じクロック周波数であっても、最新 CPU の方が 1 クロックあたりに処理できる仕事量が大きい。

##### ＜メモリアクセスとキャッシュ＞

CPU の演算器に対して、メモリへのアクセスは遅い。そのため、CPU の性能はメモリ待ちによって大きく左右される。

昔の CPU にもメモリアクセスを高速化する工夫はあった。
Intel486 DX2 は、8KB のコード・データ兼用オンチップキャッシュ、64/80MB/s のバーストバス、二次キャッシュのサポートを備えていた。(https://ardent-tool.com/CPU/docs/Intel/486/datasheets/271293-002.pdf)

キャッシュの有無ではなく、メモリ待ちをどれだけ隠せるかが進化した。

Pentium Pro では、アウトオブオーダ実行と投機実行により、ある命令がメモリアクセスで待っている間にも、依存していない別の命令を先に進められるようになった。
Pentium Pro の性能解析論文でも、アウトオブオーダ・投機実行エンジンが、保留中のメモリアクセスと有用な処理を重ねて実行し、キャッシュミスの影響を減らすと説明されている。(https://tams.informatik.uni-hamburg.de/lehre/2001ss/proseminar/mikroprozessoren/papers/pentium-pro-performance.pdf)

さらに現代 CPU では、この仕組みがより大きく発展している。
AMD Zen 5 では、L1 データキャッシュが 48KB・12-way になり、同じ種類の場所に割り当てられるデータをより多く保持できるようになっている。
さらに、ロード・ストア用の処理経路が増え、ロードキューやストアキューも大きくなったことで、複数の未完了メモリアクセスを同時に管理できるようになった。つまり、単にキャッシュ容量を増やすだけでなく、複数のメモリアクセスを同時に抱え、予測できるアクセスを先に取りに行き、待ち時間の間に他の命令を進める能力が強化されている。(https://hc2024.hotchips.org/assets/program/conference/day2/24_HC2024.AMD.Cohen.Subramony.final.pdf)

そのため、メモリが遅いという問題を完全になくすことはできないが、待ち時間を隠す能力が昔よりはるかに高い。メモリアクセスに関する性能差は、次のように整理できる。

Intel486 DX2 では、オンチップキャッシュやバーストバスによって、メモリアクセスそのものを高速化しようとしていた。

Pentium Pro では、アウトオブオーダ実行によって、メモリ待ちの間にも依存していない命令を進められるようになった。

現代 CPU では、多段キャッシュ、プリフェッチャ、ロード・ストアキュー、多数の未完了メモリアクセスの追跡によって、メモリ待ちをさらに隠せるようになっている。

##### ＜投機実行＞

現代 CPU は、まだ確定していない処理も予測に基づいて先に実行する。
予測が当たれば、待ち時間がなくなる。予測が外れた場合は、投機的に実行した結果を捨て、正しい経路からやり直す。

Pentium Pro / P6 世代では、この考え方が Dynamic Execution として本格的に導入された。Intel の資料では、Dynamic Execution は分岐予測、投機実行、データフロー解析を組み合わせた仕組みであり、Pentium Pro の高性能化の鍵として説明されている。(https://tams.informatik.uni-hamburg.de/lehre/2001ss/proseminar/mikroprozessoren/papers/pentium-pro-performance.pdf)

最新 CPU では、この仕組みがさらに発展している。たとえば AMD Zen 5 の発表資料では、分岐予測について「より少ないバブル」「より高い精度」「より高いスループット」が示され、より大きな TAGE 系予測器、16K エントリ規模の L1 BTB、52 エントリの return address stack、1 サイクルあたり 2 個の taken branch 予測などが説明されている。これにより、分岐によって命令供給が止まる場面を減らし、結果として CPI の低下に寄与する。(https://hc2024.hotchips.org/assets/program/conference/day2/24_HC2024.AMD.Cohen.Subramony.final.pdf)

### Q2.3

> 分岐予測、投機的実行、Out-of-Order実行の中から1つ選んで、どのような技術かを簡単に説明してください。また、その動作を観測するプログラムを書いて、プログラムと実行環境、結果と考察を教えてください。観測できなかった場合は、なぜ観測できなかったか、どうすれば観測できそうかを考察してください。

#### 回答

分岐予測を選んだ。

##### ＜簡単な説明＞

CPU は if 文のような条件分岐に出会ったとき、本来は条件の結果が確定するまで次に進む命令を決められない。
しかし毎回そこで待つとパイプラインが止まり、性能が落ちる。
そこで CPU は過去の分岐結果などから「次はこっちに進むだろう」と予測し、予測した先の命令を先に実行する。
これが分岐予測である。

予測が当たれば高速に処理を続けられる。
一方、予測が外れると、先に進めていた処理を捨てて正しい分岐先からやり直す必要がある。
そのため、予測しにくい分岐は予測しやすい分岐より遅くなることがある。

##### ＜観測方法＞

分岐予測を観測するためのプログラムを作成した。プログラムは添付欄に添付した `main.c` である。

このプログラムでは、同じループ回数で次の 4 種類の入力を処理する。プログラムの引数は `0`、`1`、`2`、`3` で指定する。

- `0`: 配列の値をすべて `0` にする。分岐結果が毎回同じなので予測しやすいはずである。
- `1`: 配列の値を `0, 1, 0, 1, ...` と交互にする。単純な規則があるので、ランダムよりは予測しやすい可能性があると仮定した。
- `2`: 配列の値を `rand()` 関数により `0` または `1` にする。分岐結果が不規則なので予測しにくいはずである。
- `3`: `2` と同じように `rand()` で配列を作成後、測定前にソート処理 (`qsort`) を行う。`0` が前半、`1` が後半に固まるので予測しやすくなるはずである。

ループ内では、次のような分岐を行う。

```c
if (data[i]) {
    case_true(i);
} else {
    case_false(i);
}
```

`case_true` と `case_false` は `noinline` にしている。
これは、コンパイラ最適化によって分岐が消えたり、条件選択命令だけに変換されたりするのを避けるためである。

なお、配列生成やソートは測定前に行っている。そのため、`rand()` や `qsort()` そのものの実行時間は測定結果に含まれない。

```c
{
	unsigned long x;
	asm volatile ("csrr %0, 0x7c1" : "=r"(x));
	return x;
}

static inline void duo_write_mhcr(unsigned long x)
{
	asm volatile ("csrw 0x7c1, %0" :: "r"(x) : "memory");
}

static void duo_disable_branch_predictor(void)
{
	unsigned long x = duo_read_mhcr();

	x &= ~(MHCR_RS | MHCR_BPE | MHCR_BTB);

	duo_write_mhcr(x);

	asm volatile ("fence.i" ::: "memory");
}
```

これを `rc = sbi_console_init(scratch);` が呼ばれている後の行で呼び出した。
具体的に変更後は以下のようにした。

##### ＜考察＞

実験結果1では、`2`、つまり配列がランダムな場合が最も遅く、`0`、つまり配列がすべて 0 である場合が最も速かった。
これは、分岐結果が不規則な場合に分岐予測が外れやすくなり、実行時間が伸びたためだと考えられる。
また、`2` よりも `3` のほうが速い点は重要である。`3` は `rand() & 1` で作ったデータをソートしているので、0 と 1 の個数は `2` とほぼ同じである。しかし、並び順は前半が 0、後半が 1 という形に近くなる。そのため、分岐結果に規則性が生まれ、分岐予測がしやすくなったと考えられる。

実験結果2を見ると、実験結果1で観察された差がほぼ消えていたことがわかる。
つまり、実験結果1の差は分岐予測によるものであった可能性が高い。

特に、最も分岐結果が不規則である `2` が、他の条件とほとんど変わらない時間になっている。

また、気になる点もある。
今回は全体的に前回より遅くなっている。特に `0` は 3.140 sec から 3.934 sec へ大きく遅くなった。
これは、予測しやすい分岐で得られていた利益が消えた、と見ると自然である。
ただし、ファームウェア変更に伴うクロック、カーネル設定、メモリ設定、割り込み負荷など別の要因も混ざる可能性はある。

したがって、今回の実験だけでレジスタ設定が意図通りに効いたと完全に断定することはできない。
しかし、分岐予測が有効な環境で見えていた `0` と `2` の差、および `2` と `3` の差が、分岐予測を無効化した環境ではほぼ消えている。
このため、実行時間の傾向としては、分岐予測を無効化する変更が効いていることを強く示していると考えられる。

結論として以下を示す。

この実験では、分岐予測が有効な環境では、予測しにくいランダムな分岐が最も遅くなった。また、同じランダム由来のデータでも、ソートして分岐結果の並びを規則的にすると速くなった。

一方、MHCR の BPE と BTB を落とした環境では、入力パターンによる実行時間差がほぼ消えた。この結果から、分岐予測の有無が実行時間に影響していることを観測できたと考えられる。

```c
#include <stdint.h>
#include <stdio.h>
#include <stdlib.h>
#include <time.h>

#define N (1 << 20)
#define R (80)

static volatile uint64_t tmp;
int data[N];

int cmp(const void *a, const void *b) {
  return *(const int *)a - *(const int *)b;
}

double current_sec(void) {
  struct timespec ts;
  clock_gettime(CLOCK_MONOTONIC, &ts);
  return ts.tv_sec + ts.tv_nsec * 1e-9;
}

__attribute__((noinline)) void case_true(int i) { tmp += (uint64_t)i * 3 + 1; }

__attribute__((noinline)) void case_false(int i) { tmp += (uint64_t)i * 7 + 1; }

void measure(int mode) {
  // fill data
  srand(1);
  for (int i = 0; i < N; i++) {
    if (mode == 0) {
      data[i] = 0;
    } else if (mode == 1) {
      data[i] = i & 1;
    } else {
      data[i] = rand() & 1;
    }
  }
  if (mode == 3) {
    qsort(data, N, sizeof(data[0]), cmp);
  }
  tmp = 0;
  double start = current_sec();
  // run
  for (int r = 0; r < R; r++) {
    for (int i = 0; i < N; i++) {
      if (data[i]) {
        case_true(i);
      } else {
        case_false(i);
      }
    }
  }
  double end = current_sec();
  printf("%.6f sec\n", end - start);
}

int main(int argc, char **argv) {
  if (argc == 2) {
    int mode = atoi(argv[1]);
    if (0 <= mode && mode <= 3) {
      measure(mode);
    }
  }
  return 0;
}
```

### Q2.4

> 任意のISAの中から定義を1つ選んで、そのように定義されている理由や経緯を教えてください。その定義を変えればCPUを高速化できそうかを考察してください。ISAはあなたしか知らない自作のISAでも構いませんが、その場合は仕様書にアクセスする方法を教えてください。

#### 回答

ISA として RISC-V を選び、その中で RVWMO (RISC-V Weak Memory Ordering) という定義を選んだ。
これは、RISC-V におけるメモリ一貫性モデル、つまり複数のハードウェアスレッド (hart) が共有メモリを読み書きするときに、ロードやストアの結果がどのような順序で見えてよいかを決める ISA 上の規則である。
RISC-V 公式仕様では、メモリ一貫性モデルは「ロードが返してよい値を定める規則」と説明され、RISC-V は標準で RVWMO という弱いメモリモデルを採用している。

RVWMO では、ある hart の中では命令が順番に実行されているように見えるが、別の hart から見ると、その hart のメモリアクセスがプログラム順とは異なる順序で観測される場合がある。
そのため、マルチスレッドプログラムでメモリアクセス順序を保証したい場合には、FENCE 命令や、atomic 命令の acquire、release 指定などを使って明示的に同期する必要がある。

RVWMO が弱いメモリモデルとして定義されている理由は、CPU 実装に高い自由度を与えるためである。
現代の高性能 CPU は、命令を必ずプログラムに書かれた順番で実行しているわけではない。
ロードを先に実行したり、ストアをストアバッファに一時的に保持したり、依存関係のないメモリアクセスを並べ替えたりすることで、メモリ待ち時間を隠して性能を上げている。

もし ISA が「すべてのメモリアクセスは、すべての hart から常にプログラム順に見えなければならない」と定義してしまうと、CPU はこのような最適化をしにくくなる。
そこで RISC-V では、基本仕様では RVWMO という比較的弱いモデルを採用し、ハードウェアが高性能・高スケーラビリティな実装を作れるようにしている。
公式仕様も、RVWMO は「高性能でスケーラブルな設計を構築する柔軟性」と「扱いやすいプログラミングモデル」を両立させるためのものだと説明している。(https://docs.riscv.org/reference/isa/unpriv/rvwmo.html)

また、RISC-V 公式の解説資料では、弱いメモリモデルはハードウェア実装の柔軟性、性能、電力効率、スケーラビリティの点で有利になりやすい一方、プログラミングモデルは複雑になると説明されている。
逆に、強いメモリモデルはプログラマには分かりやすいが、パイプラインやメモリシステムで行える最適化を制限し、電力・面積・検証負担にもコストを課すとされる。RISC-V はその中間的な位置として RVWMO を選んでいる。(https://docs.riscv.org/reference/isa/unpriv/mm-eplan.html)

RVWMO が CPU 性能に関係するのは、主に 命令の発行、実行、メモリアクセス、コミット の段階である。
命令のフェッチやデコードを単純化する話ではなく、ロード命令やストア命令をCPU内部でどこまで自由に処理できるかに関係する。

たとえば、アウトオブオーダ実行を行う CPU では、先に実行できるロードをプログラム順より早く実行したい場合がある。
また、ストアはすぐに共有メモリへ反映せず、ストアバッファに入れて後で処理したい場合がある。
RVWMO のような弱いメモリモデルでは、プログラムが明示的に順序を要求していない限り、こうした並べ替えや遅延が許されやすい。
これにより、CPU はメモリ待ち時間を隠しやすくなり、複数のメモリアクセスを効率よく並行処理できる。

一方で、FENCE 命令や acquire、release 指定がある場合、CPU はその前後のメモリアクセス順序を守る必要がある。
そのため、フェンスが多いプログラムでは、CPU 内部の並べ替え自由度が減り、性能が落ちる可能性がある。
つまり、RVWMO は「何でも自由に並べ替えてよい」という定義ではなく、必要なところだけ同期を明示し、それ以外では CPU に自由度を与える設計だと言える。

RVWMO を変えれば CPU が高速化できるかどうかは、どちらの方向に変えるかによって答えが変わると考察する。

まず、RVWMO をより強いメモリモデルに変える場合を考える。
たとえば、x86 に近い TSO (Total Store Ordering) のようなモデルを RISC-V の標準にすると、プログラマにとっては理解しやすくなる。
既存の x86 向け並行プログラムを RISC-V へ移植する場合にも楽になる可能性がある。
実際、RISC-V には Ztso という拡張が用意されており、これは x86 や SPARC のように TSO を前提とするコードの移植を助けるためのものと説明されている。
(https://docs.riscv.org/reference/isa/unpriv/ztso-st-ext.html)

しかし、CPU の高速化という観点では、RVWMO を TSO へ強めることは一般に有利とは言いにくい。
強いメモリモデルにすると、CPU がロードやストアを並べ替えられる範囲が狭くなるからである。
Ztso の仕様でも、ロードは acquire 相当、ストアは release 相当として扱われ、RVWMO より強い順序制約を与えると説明されている。
したがって、TSO 化は「ソフトウェアを分かりやすくする変更」ではあっても、「ハードウェアをより高速にする変更」とは限らない。

逆に、RVWMO よりさらに弱いメモリモデルに変える場合を考える。
この場合、CPU はより多くのロード・ストアを自由に並べ替えられる可能性があり、理論上はさらに高速化できる余地がある。
特に、多数のコアが共有メモリにアクセスする環境では、メモリ順序制約が少ないほど、ストアバッファやメモリ依存予測を積極的に使いやすい。

しかし、その代償として、プログラミング、コンパイラ、OS、検証が難しくなる。
どこにフェンスを入れるべきかを正しく判断するのが難しくなり、並行プログラムのバグも増えやすくなる。
RISC-V の解説資料でも、弱いモデルは性能面では有利になりやすい一方、プログラミングモデルを複雑にすると説明されている。
(https://docs.riscv.org/reference/isa/unpriv/mm-eplan.html)

したがって、RVWMO はかなり意識的な折衷案だと考えられる。つまり、RISC-V は基本仕様では CPU 実装の自由度を重視して RVWMO を採用し、必要に応じて Ztso でより強い順序モデルも選べるようにしている。この設計により、高性能実装とソフトウェア移植性の両方に対応している。

まとめると、この定義を変えれば CPU を高速化できるかという問いに対しては、単純に「はい」とは言えない。
RVWMO を TSO のように強くすれば、プログラマには分かりやすくなるが、CPU の最適化自由度は減るため、高速化にはつながりにくい。
逆に、RVWMO よりさらに弱くすれば、CPU 側の自由度は増えるかもしれないが、ソフトウェアの正しさを保つのが難しくなる。

よって、RVWMO は「CPU を速くするために弱くした定義」というより、CPU 実装の自由度と、ソフトウェアが扱える程度の分かりやすさを両立するための定義であると言えると思う。
RISC-V の現在の設計、つまり RVWMO を基本とし、必要なら Ztso を使える という形は、性能と可搬性のバランスを取った合理的な設計だと考えられる。

### Q2.5

> 今までに作ったり、書いたことがあるCPU(シミュレータや周辺部品でも可)について、そのテスト手法を教えてください。テストは何を対象にして何を検証したのか、今後テスト手法を改善するならどうするかを書いてください。まだCPUを作ったことがない場合は、CPUをハードウェア記述言語で書くにあたって、どのように設計してテストするかの計画を立てて教えてください。

#### 回答

Q1.1 でも触れた通り、RISC-V CPU を自作したことがある。
Ultra96-V2 ボード上で動作することを目標に実装した RV64IMAC_Zicsr_Zicntr 相当の CPU コアで、UART やメモリマップド I/O、画面描画用のコマンド処理も扱えるようにした。

なお、これは大学の授業の自由課題として取り組んだものである。
描画コマンドを処理する IP のコードはそれ自体が演習の解答となってしまうため、公開リポジトリには含まれていない。

テストでは、大きく分けて 命令セットレベルのテスト と Vivado シミュレータを用いた SoC としての動作確認 を行った。
命令セットレベルのテストでは、RISC-V の標準的なテストである riscv-tests を使用した。
具体的には、rv64ui、rv64mi、rv64um、rv64ua、rv64uc などのテストプログラムをビルドし、それらを HEX 形式に変換して CPU の命令メモリに読み込ませた。
リポジトリ内にも、riscv-tests を Docker でビルドし、ELF/BIN/HEX 形式へ変換して test/hex/riscv-tests に出力する仕組みがある。

このテストの対象は、主に CPU コアの命令実行機能である。
整数演算命令、分岐命令、ロード・ストア命令、例外・割り込みに関係する命令、乗除算命令、アトミック命令、圧縮命令などが、RISC-V の仕様通りに実行できるかを検証した。
各テストプログラムを CPU 上で実行し、テスト用の終了判定領域である tohost への書き込みや、テストベンチ側のログ出力をもとに、成功・失敗を判定した。

Vivado シミュレータによるテストでは、CPU 単体だけでなく、周辺回路を含めた動作も確認した。
この CPU は、Ultra96-V2 上で動かすことを想定しているため、単に ALU やデコーダだけを個別にテストするのではなく、トップモジュール、メモリ、UART、AXI VIP、レジスタバス、MMIO などを含めた形でシミュレーションプロジェクトを作成した。
simulation.tcl では、Ultra96-V2 に対応する Zynq UltraScale+ の part/board 設定を行い、生成済み RTL、Boot ROM、テストベンチ、HEX ファイル、FIFO IP、AXI VIP などを Vivado プロジェクトに追加している。

一方で、現在のテスト手法には改善できる点もある。まず、riscv-tests は命令単位の基本的な正しさを確認するには有効だが、ランダムな命令列や複雑なパイプラインハザード、割り込みタイミング、キャッシュやメモリ待ち時間を含むようなケースを十分に網羅できるとは限らない。
そのため、今後は riscv-dv のようなランダム命令生成テストを導入し、Spike や QEMU などの参照実装と実行結果を比較する差分テストを行いたい。
これにより、人間が手で考えたテストだけでは漏れやすい命令列や境界条件を検出しやすくなると考えている。

次に、現在はシミュレーションログの文字列を見て PASS、FAIL を判断しているが、今後はテスト結果をより機械的に扱えるようにしたい。
たとえば、各テストの命令数、サイクル数、終了理由、失敗時の PC、直前の命令、レジスタファイルの内容、CSR の値などを自動で保存し、失敗原因を追いやすくすることが考えられる。
加えて、GitHub Actions などの CI に組み込み、コミットごとに基本的な riscv-tests と lint を自動実行できるようにすれば、モダンだったのかなと感じている。

また、描画コマンドを処理する IP についても、CPU コアと同様に単体でテストを行った。この IP は、CPU から送られてくる描画命令を受け取り、画面に表示するためのピクセルデータを生成する部分である。そのためテストでは、描画命令を正しく解釈できるか、指定した座標・色・範囲に対して期待通りのピクセルデータが生成されるかを検証した。

シミュレーションでは、実際に描画命令を入力として与え、IP がディスプレイへ送る予定のデータを txt ファイルとしてダンプするようにした。さらに、その txt ファイルを画像として表示する補助プログラムを作成し、描画結果を視覚的に確認できるようにした。これにより、たとえば指定した位置に正しい色のピクセルが表示されるか、矩形や文字などの描画範囲がずれていないか、画面端付近の描画で範囲外アクセスや欠けが起きないかを確認した。

このテストは、単に波形を見るだけでは分かりにくい描画処理の正しさを確認する上で有効だった。特に画面描画では、1 ピクセル単位の座標ずれや色の取り違えが波形上では直感的に把握しにくいため、出力を画像として復元して確認できるようにしたことで、バグの原因を追いやすくなった。

一方で、この方法は最終的な確認を目視に頼る部分が残っていたため、今後改善するなら、生成された画像と期待画像を自動比較する仕組みを追加したい。たとえば、テストごとに期待されるフレームバッファ画像を用意しておき、シミュレーション結果から生成した画像との差分を取ることで、ピクセル単位で一致しているかを自動判定できるようにする。また、完全一致だけでなく、差分が発生した座標や色をログとして出力すれば、描画 IP の不具合をより効率的に調査できる。
